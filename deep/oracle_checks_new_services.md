# Новые проверки-оракулы для T-Invest API (UsersService, стримы, кросс-сервис)

Ниже — набор новых проверок-оракулов для T‑Invest API, дополняющих уже существующие по сервисам `MarketData`, `Instruments`, `Operations`, `Orders`, `Sandbox`, `StopOrders`. Я специально делаю акцент на:

- новые сервисы (`UsersService`, стрим‑сервисы `MarketDataStreamService`, `OperationsStreamService`, `OrdersStreamService`);
- кросс‑сервисные инварианты, которых обычно нет в «локальных» проверках по одному сервису.

Для каждой проверки указываю: **объективный оракул**, **инвариант**, **частоту**, **риск лимитов**, **тип ловимых ошибок**.

---

## 1. UsersService: согласованность счетов и маржинальных атрибутов

**Цель**: убедиться, что UsersService отдает полный и согласованный с остальными сервисами и UI набор данных о счетах и марже.

**Объективный oracle**

- Список счетов пользователя в веб‑/мобильном интерфейсе Т‑Банк (канонический источник).
- Расчет маржинальных показателей на стороне тестов по открытым позициям (через `OperationsService.GetPositions` + котировки из `MarketDataService`).

**Ожидаемый инвариант**

1. Список счетов из `UsersService.GetAccounts`:
   - содержит все и только те счета, которые видит пользователь в личном кабинете;
   - статусы («открыт»/«закрыт»), типы (брокерский/ИИС) и валюты совпадают с UI.
2. Суммарная стоимость портфеля и маржинальные показатели из `UsersService.GetMarginAttributes` согласованы с позициями:
   - `equity ≈ Σ(позиция_i * цена_i)` по `GetPositions` (погрешность — только на округление).

**Частота запуска**

- 1 раз в день после завершения торговой сессии.
- Дополнительно — при событии изменения списка счетов (открытие/закрытие счета в тестовом профиле).

**Риск API‑лимитов**

- Низкий: 1–2 запроса (`GetAccounts`, `GetMarginAttributes`) + 1 запрос на позиции/котировки при необходимости.

**Какие ошибки ловит**

- Несоответствие списка счетов между API и фронтами.
- Некорректный расчет маржи (позиции не учтены, дублированы, неверные курсы).
- «Зомби‑счета» (закрытый счет продолжает приходить как активный).

## 2. UsersService: корректность тарифных лимитов (GetUserTariff)

**Цель**: проверить, что заявленные тарифные лимиты действительно отражают и описывают реальное ограничение API.

**Объективный oracle**

- Тарифный план из клиентского договора или официального описания тарифа на сайте (лимиты по RPS, доступные сервисы, число стримов).

**Ожидаемый инвариант**

1. Значения лимитов в `UsersService.GetUserTariff` совпадают с описанием тарифа.
2. При искусственном превышении лимита:
   - API возвращает `RESOURCE_EXHAUSTED`/аналогичный код, а не 500;
   - содержимое ошибки содержит информацию о сбросе лимита (reset time).
3. Заголовки/метаданные с остатком лимита (если есть) уменьшаются предсказуемо и сбрасываются по расписанию.

**Частота запуска**

- 1 раз в неделю на реальном тарифе.
- Ежечасно/ежедневно — нагрузочные тесты только в Sandbox.

**Риск API‑лимитов**

- Средний: проверка требует контролируемо «упереться» в лимит. Поэтому:
  - запускать преимущественно в песочнице;
  - в продакшене — только редкие точечные проверки с большим запасом до лимита.

**Какие ошибки ловит**

- Реальные лимиты не совпадают с задекларированными (занижены/завышены).
- Некорректная обработка превышения лимита (500 вместо `RESOURCE_EXHAUSTED`, блокировка токена).
- Баги в подсчете/сбросе лимитов (остаток не падает, не сбрасывается, «зацикленный» лимит).

## 3. UsersService: корректность внутренних переводов (CurrencyTransfer)

**Цель**: гарантировать, что переводы между счетами через API проходят атомарно и корректно отражаются в операциях и балансах.

**Объективный oracle**

- Состояние балансов и операций до и после перевода по обоим счетам (через `OperationsService`, `GetPositions`).
- Модельный баланс, считаемый в тесте (старый баланс – списание + зачисление – комиссия).

**Ожидаемый инвариант**

1. Успешный `CurrencyTransfer` создает:
   - одну операцию списания на исходном счете;
   - одну операцию зачисления на целевом счете;
   - суммы зеркальны (с точностью до комиссии).
2. Никакой «третьей» операции или расхождения остатков балансов быть не должно.
3. Попытка перевода между счетами в разных валютах без явной конвертации порождает валидационную ошибку (`INVALID_ARGUMENT` / аналог).

**Частота запуска**

- Раз в неделю в Sandbox с фиксированным тестовым юзером.
- Раз в месяц — на тестовом реальном профиле с малыми суммами (или только песочница, если политика безопасности строгая).

**Риск API‑лимитов**

- Низкий: единичный запрос перевода + 1–2 запроса на проверки балансов/операций.

**Какие ошибки ловит**

- Потеря денег при переводах (списание без зачисления).
- Дублирование операций (двойное списание/зачисление).
- Ошибка комиссий и работы с разными валютами (например, перевод RUB→USD без пересчета).

## 4. MarketDataStreamService: непротиворечие стрима и unary‑котировок

**Цель**: убедиться, что стрим котировок не расходится с каноничными unary‑методами `MarketDataService`.

**Объективный oracle**

- Ответ `MarketDataService.GetLastPrices` по тем же FIGI/тикерам и в близкий момент времени.

**Ожидаемый инвариант**

- Для каждой бумаги:
  - `|price_stream − price_unary| / price_unary ≤ 0.0005` (0.05%) **или** расхождение ≤ 1 минимального шага цены.
  - При резком движении рынка (визуально по свечам) расхождение не держится дольше N секунд (напр. 2–3 с).

**Частота запуска**

- Для топ‑20 ликвидных инструментов: каждые 5 секунд в торговую сессию.
- Для расширенного списка: каждые 5–10 минут.

**Риск API‑лимитов**

- Средний: ~12 запросов в минуту для топ‑20 (при 10 RPS допустимо, но требует дисциплины и пуллинга по пачке FIGI).

**Какие ошибки ловит**

- Отставание стрима от реальности или от unary‑API (деградация системы стриминга).
- Баги маппинга идентификаторов (стрим шлет цену не той бумаги).
- Порча или некорректная десериализация данных стрима.

## 5. OperationsStreamService: полнота и корректность потока операций (PortfolioStream)

**Цель**: проверить, что все изменения портфеля и операций, видимые в батч‑методах, появляются и в стриме, без потерь и дубликатов.

**Объективный oracle**

- `OperationsService.GetOperations` и/или `GetPortfolio` за тот же временной интервал и по тем же счетам.

**Ожидаемый инвариант**

- За выбранный период:
  - множество операций, пришедших через `OperationsStreamService.PortfolioStream`, биеквивалентно множеству операций из `GetOperations`;
  - нет операций, которые есть в батче, но ни разу не пришли в стриме;
  - нет «лишних» операций в стриме, отсутствующих в `GetOperations`;
  - по каждой операции совпадают ключевые поля: тип, инструмент, количество, сумма, время (с допуском по времени 1–2 секунды).

**Частота запуска**

- 1 раз в день после закрытия сессии (сверка за весь день).
- Дополнительно: выборочные часовые окна, если есть подозрения на деградацию стримов.

**Риск API‑лимитов**

- Низкий: 1–2 вызова `GetOperations` на нужное окно, стрим не расходует лимит запросов.

**Какие ошибки ловит**

- Потеря или дублирование событий в стриме.
- Неконсистентность данных (в стриме и батче разные суммы, типы операций).
- Ошибки миграции формата событий или фильтрации.

## 6. OrdersStreamService: валидность жизненного цикла заявок

**Цель**: убедиться, что стрим заявок (`OrdersStreamService.OrderStateStream`) следует корректному конечному автомату статусов.

**Объективный oracle**

- Формальное описание допустимых переходов статусов в документации T‑Invest API.

**Ожидаемый инвариант**

- Для каждой заявки последовательность статусов, пришедшая по `OrderStateStream`, удовлетворяет разрешенному графу переходов:
  - нет переходов `Executed → New`, `Executed → Working`, `Rejected → Working` и т.п.;
  - не бывает двух разных финальных статусов (например, и `Executed`, и `Canceled`);
  - нет «отката» в более ранний статус (после `PartiallyExecuted` нельзя вернуться в `New`).

**Частота запуска**

- Постоянно, в онлайне: проверки прямо в обработчике событий стрима.

**Риск API‑лимитов**

- Нулевой: анализ выполняется на стороне клиента по уже полученным событиям.

**Какие ошибки ловит**

- Race condition’ы в движке заявок.
- Некорректные ручные правки/миграции статусов в БД.
- Ошибки в обновлении контрактов/enum’ов статусов.

## 7. OrdersStreamService + OperationsService: соответствие сделок и операций

**Цель**: гарантировать, что все сделки, пришедшие по `TradesStream`, корректно отражены в операциях и позициях.

**Объективный oracle**

- `OrdersStreamService.TradesStream` (факт сделок в онлайне).
- `OperationsService.GetOperations` (набор операций типа «сделка»).

**Ожидаемый инвариант**

- Для каждой сделки из `TradesStream` есть ровно одна операция в `GetOperations` с:
  - тем же FIGI/тикером;
  - тем же количеством, направлением (buy/sell);
  - той же ценой (с точностью до округления).
- Не существует операции «сделка», которой нет в истории `TradesStream` за тот же период (при условии, что стрим был подписан непрерывно).

**Частота запуска**

- Раз в час в торговую сессию (скользящее окно).
- Полный аудит — раз в сутки.

**Риск API‑лимитов**

- Низкий: 1 запрос `GetOperations` на часовой слот.

**Какие ошибки ловит**

- Потерю событий сделок в стриме.
- Ошибки записи операций (разные суммы/количества в стриме и в батче).
- Дубли операций или сделок.

## 8. InstrumentsService + MarketDataService: консистентность расписаний торговых сессий

**Цель**: убедиться, что расписание торговых сессий и фактические данные по свечам не противоречат официальным биржевым календарям.

**Объективный oracle**

- Официальные производственные календари соответствующих бирж (MOEX, NYSE, NASDAQ и др.).

**Ожидаемый инвариант**

1. Расписание `InstrumentsService.GetTradingSchedules` совпадает с расписанием биржи:
   - торговые/неторговые дни;
   - время начала и окончания сессий.
2. Запрос исторических свечей `MarketDataService.GetCandles` за явно неторговые дни возвращает пустой набор данных.
3. В интервалах, когда торги еще не начались или уже окончены, новые свечи не появляются.

**Частота запуска**

- Ежедневно перед стартом торгов (проверка текущего дня).
- Раз в неделю — сверка расписания на следующую неделю.

**Риск API‑лимитов**

- Низкий: 1–2 запроса расписаний и несколько выборочных запросов свечей.

**Какие ошибки ловит**

- Неправильные/устаревшие сведения о праздниках и сокращенных сессиях.
- Появление «фантомных» свечей в нерабочие дни.
- Ошибки таймзон (MOSCOW vs UTC).

## 9. InstrumentsService + OperationsService: корректность дивидендов (кросс‑сервисная проверка)

**Цель**: проверить, что справочная информация о дивидендах и реальные начисления по счетам согласованы и совпадают с данными эмитента.

**Объективный oracle**

- Официальные корпоративные действия и объявления эмитента/биржи (дивиденды: ex‑date, amount, currency, payment date).

**Ожидаемый инвариант**

1. `InstrumentsService.GetDividends`:
   - содержит все объявленные дивиденды по инструменту;
   - значения дат и сумм совпадают с первоисточником.
2. На счету, владевшем бумагой на дату отсечки, появляется операция «дивиденд» в `GetOperations`:
   - сумма = количество акций на отсечку × дивиденд на акцию;
   - дата зачисления в разумном окне после payment date.

**Частота запуска**

- Еженедельно в «дивидендный сезон».
- Ежемесячно вне пиковых периодов.

**Риск API‑лимитов**

- Низкий: выборочный список бумаг + 1–2 запроса `GetDividends` и `GetOperations`.

**Какие ошибки ловит**

- Ошибки выгрузки/обработки корпоративных действий в InstrumentsService.
- Отсутствие начислений по дивидендам (фактически украден доход клиента).
- Ошибки валюты/dividend gross vs net, неправильный пересчет у иностранных бумаг.

## 10. SandboxService + InstrumentsService: паритет песочницы и продакшена по инструментам

**Цель**: обеспечить, чтобы сценарии, отлаженные в песочнице, были валидны в продакшене (одинаковые инструменты и их свойства).

**Объективный oracle**

- Список торгуемых инструментов и их параметры в продакшене (`InstrumentsService` на прод‑endpoint).
- Документация по Sandbox (заявленные отличия, если они есть).

**Ожидаемый инвариант**

- Множество FIGI/тикеров в Sandbox — подмножество множества в продакшене.
- Для общих FIGI:
  - совпадает валюта, лот, шаг цены, доступность к торгам;
  - торговые расписания идентичны (за исключением явно задокументированных учебных отличий).
- Не существует «sandbox‑only» инструментов, не упомянутых в документации.

**Частота запуска**

- После каждого релиза, затрагивающего инструменты (автоматически по pipeline).
- Плановый аудит — раз в месяц.

**Риск API‑лимитов**

- Низкий: по одному вызову для списка инструментов в Sandbox и в продакшене (с возможной пагинацией, но без высокой частоты).

**Какие ошибки ловит**

- Рассинхронизацию инструментов между тестовой и боевой средой.
- Непреднамеренное появление тестовых/закрытых инструментов в Sandbox.
- Ломку тестов после релиза из‑за незаметных изменений параметров инструмента.

---

## Итог

Эти проверки:

- покрывают **новые сервисы** (`UsersService`, стримы рынка/операций/заявок), которые не попадают в ваши текущие проверки по шести основным сервисам;
- добавляют **критичные кросс‑сервисные инварианты** (дивиденды, расписания, сделки ↔ операции, sandbox ↔ prod);
- в большинстве случаев имеют **низкий риск по лимитам**, а потенциально «тяжелые» (лимиты тарифа, сверка стрима с unary) можно запускать:
  - либо в песочнице,
  - либо с ограниченным набором инструментов/счетов и контролируемой частотой.
