# Согласованность портфеля и маржи — account-specific оракулы

**Дата:** 2026-06-19 · `analysis/portfolio_margin_validate.py` · READ-ONLY · счета брокерский и копилка

> Числа этого прогона (стоимость портфеля, маржа, позиции) относятся к реальному счёту автора
> и не публикуются — здесь только методология и проверенные соотношения.

## Задача
Проверить, что поля API по конкретному счёту внутренне согласованы — тождествами, которые
обязаны выполняться, если стоимость, P&L и маржа считаются непротиворечиво. Оракулы — сами
поля API (`GetPortfolio`, `GetMarginAttributes`).

## Точные тождества маржи (главный результат, без внешних цен)
На брокерском счёте все три выполняются **до копейки** (Δ=0.0000₽):
- **D** `liquidPortfolio == totalAmountPortfolio`
- **E** `minimalMargin == startingMargin / 2`
- **F** `amountOfMissingFunds == startingMargin − liquidPortfolio` (отрицательное значение
  здесь означает не дефицит, а запас обеспечения: ликвидный портфель кратно покрывает
  начальную маржу)

Копилка — маржинальная торговля отключена (`Account margin status is disabled`), тождества
D/E/F к ней неприменимы — это тоже зафиксированный факт.

## Реконструкция стоимости и P&L из позиций
- **A. `totalAmountPortfolio == Σ qty·текущая_цена`** — на брокерском счёте сходится точно
  (кэша нет). На копилке — расхождение на уровне долей копейки на единицу: дрейф снимка
  цены (портфель оценён сервером в момент T1, `GetMarketValues` — в T2≠T1).
- **C. `expectedYield(%) == 100·P&L/вложено_fifo`** — сходится с точностью до округления
  поля до 2 знаков.
- P&L по позиции (`expectedYieldFifo`) сходится с независимым расчётом `Σ qty·(цена−fifo)`
  до округления `averagePositionPriceFifo` до 2 знаков (расхождение на уровне долей копейки
  на единицу — показанная средняя цена округлена, истинная чуть точнее).

## Вскрытая грабля: риск-ставки не публикуются для биржевых фондов
`GetRiskRates` по позициям (TMON@, и инструменты копилки) возвращает **ПУСТЫЕ** `longRiskRates`/
`shortRiskRates`. Поэтому начальную маржу `startingMargin` НЕЛЬЗЯ пересчитать из публикуемой
риск-ставки напрямую. Косвенно ставка вычисляется как `startingMargin / стоимость_позиции` —
но это вывод из уже посчитанной маржи, а не независимый источник. Вывод: для расчёта
покупательной способности по таким фондам полагаться на `GetMarginAttributes`, не на `GetRiskRates`.

## Практический вывод
Поля портфеля и маржи внутренне непротиворечивы и пригодны для расчёта запаса обеспечения:
`startingMargin` (полная), `minimalMargin` (= половина, уровень принудительного закрытия),
`amountOfMissingFunds` (запас/дефицит). Стоимость и P&L воспроизводятся из позиций и цен с
точностью до снимка котировки и округления средней цены до копейки.
