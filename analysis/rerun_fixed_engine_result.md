## 1. XSEC_MOMENTUM

Запуск: `python analysis/botdev_xsec_momentum.py`, дважды →
`analysis/botdev_xsec_rerun_fixed_run{1,2}.txt`.

### Проверка детерминизма: run1 ≠ run2 бит-в-бит — разобрался, причина найдена

`diff run1 run2` **не пуст**. Разница — только в блоке «grid» (полная лента,
32 испытания): для одной и той же комбинации параметров
`{lookback:126, skip:0, rebalance:21, top:2}` run1 даёт `ret +22.83% Sharpe +0.32`,
run2 — `ret +15.48% Sharpe +0.26`. Из-за этого сдвига поменялось место в топ-5
таблицы и итоговая строка «best IS» (какой конфиг оказался первым по Sharpe),
и DSR сместился на третий знак (0.496 → 0.499). **Окна walk-forward и сквозной
OOS (−22.58%) при этом идентичны run1/run2 бит-в-бит** — расхождение локализовано
строго в «grid»-прогоне на полной ленте.

Диагноз: `backtest/strategies.py:300` в `XSecMomentum.on_bar` —
`winners = {t for _, t in scored[:self.top]}` — победители собираются в `set`,
а не в список. Итерация `for t in winners: ctx.order_target_percent(...)` (строка
304) исполняет ордера в порядке, который зависит от хеша строк-тикеров, а хеш
строк в Python рандомизируется на старте процесса (`PYTHONHASHSEED` по
умолчанию не фиксирован). При равном/близком ранге на границе top-N порядок
исполнения ордеров внутри бара меняет доступный кэш для последующих
`order_target_percent` → разная итоговая доходность для ОДНИХ И ТЕХ ЖЕ входных
данных и параметров. Это **не связано с фиксом движка** (bug 1/2/3) — предсуществующий
баг воспроизводимости в коде стратегии-кандидата.

Подтверждение: `PYTHONHASHSEED=0 python analysis/botdev_xsec_momentum.py`,
запущенный дважды подряд — **бит-в-бит идентичен** (в /tmp, не сохранялось в
analysis/ по инструкции не плодить лишние файлы вне указанного списка).
Дополнительно любопытное совпадение: при `PYTHONHASHSEED=0` сквозной
warm-up OOS = **−19.85%**, DSR = 0.499 — это **дословно совпадает** с числом
«−19.85% (warm-up)» из старого отчёта `BOT_DEV_2026-07-02_REPORT.md` (вероятно,
исходный прогон отчёта тоже случайно/окружением получил тот же эффективный
порядок хеширования). Это не отменяет находку: без фиксации `PYTHONHASHSEED`
результат grid-таблицы (не WFA) для этого кандидата не воспроизводим бит-в-бит
между процессами — независимо от движка. Рекомендация (не выполнялась —
`backtest/strategies.py` вне зоны правок в этой задаче): заменить `set` на
`sorted(...)`/список на строке 300.

Вывод по детерминизму: расхождение не связано с фиксом движка, не меняет
вердикт, амплитуда укладывается в статистический шум относительно всех

## Фикс недетерминизма `XSecMomentum` (2026-07-02, вечер)

Применена рекомендация из раздела выше: `backtest/strategies.py:300` —

```python
winners = {t for _, t in scored[:self.top]}    # было: set, порядок от PYTHONHASHSEED
winners = [t for _, t in scored[:self.top]]    # стало: список, scored уже отсортирован
```

`scored.sort(reverse=True)` на предыдущей строке уже даёт список по убыванию
скора — срез `scored[:self.top]` и без `set` сохраняет порядок; проверка
`t not in winners` (используется дальше в `on_bar`) корректна и для списка.

Грепом по всему `backtest/` (не только `strategies.py`) проверены остальные
`set(...)`/set-comprehension — все безопасны для порядка ордеров:
`candles.py:227` (`seen_ts` — дедупликация меток времени данных, не порядок
исполнения), `benchmark.py:19` и `ensemble.py:32,64` (пересечение временных
осей для агрегации метрик, результат сразу оборачивается в `sorted(...)`),
`optimize.py:107` (`seen` — дедупликация точек грида) и `optimize.py:142`
(`sorted(set(...))` — уже отсортировано). Других мест, где порядок set влиял бы
на порядок ордеров внутри бара, в `backtest/` не найдено — правка ограничена
одной строкой.
