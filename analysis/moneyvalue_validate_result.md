# Конвертация MoneyValue/Quotation: units+nano против десятичной строки

**Дата:** 2026-06-19 · `analysis/moneyvalue_validate.py` · READ-ONLY · 1381 денежное поле из 5 эндпоинтов

## Задача
Денежные величины T-Invest приходят как `{units, nano}` (nano = 1e-9 доли). Проверить, что формула
проекта `units + nano/1e9` корректна, знаки согласованы, и сверить с десятичной строкой.

## Результат
По 1381 полю (Operations обоих счетов, Portfolio, BondCoupons, AccruedInterests):
- **знак units == знак nano**: 0 нарушений;
- **|nano| < 1e9**: 0 нарушений;
- **кросс-слой REST↔MCP**: `totalAmountPortfolio` = REST units 58911 + nano 4e8/1e9 = **58911.4** =
  строковое value MCP-слоя. Формула подтверждена.

## Вскрытая грабля: REST не дублирует value
Из 1381 поля **строковое поле `value` присутствовало в 0** — то есть **сырой REST-ответ НЕ кладёт
десятичную строку `value` в денежные поля**, отдаёт только `{units, nano}` (+ `currency`). Поле
`value`, на которое опираются многие парсеры и которое видно в MCP/SDK-ответах, **добавляет
обёртка** (MCP-сервер / SDK), а не сам REST.

**Практический вывод:** код, ходящий в REST напрямую (как `scripts/`, `lab/`), ОБЯЗАН парсить
`units + nano/1e9` и не рассчитывать на `value` — её там нет. Код через MCP/SDK может читать
`value`. Хелпер `to_f()` в скриптах проекта это уже учитывает (сначала пробует `value`, потом
`units/nano`), что и делает его универсальным для обоих слоёв.

## Связь с проектом
Подтверждает базовое правило из официальной документации T-Invest (custom-types) и оправдывает форму `to_f()`,
продублированную во всех скриптах: денежные величины надо собирать из units/nano, строковый
`value` — лишь удобство слоя обёртки.
