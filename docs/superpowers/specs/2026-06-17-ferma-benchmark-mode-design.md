# Свести ферму к режиму бенчмарков

Дата: 2026-06-17

## Проблема

Ферма (`lab/`, VPS, 24/7) гоняет 6 стратегий, каждую на своём sandbox-счёте 100k ₽:
контроли `buyhold`/`random` и кандидаты `grid`/`momentum`/`meanrev`/`gold_trend`.

Форвард-тест на песочнице оправдан только для стратегии, **прошедшей бэктест** — он
проверяет реальное исполнение (проскальзывание, отказы API, баги симулятора) до того,
как рисковать деньгами. Но кандидаты study НЕ прошли: у всех Deflated Sharpe = 0%
(Sharpe — артефакт перебора сетки, устойчивого OOS-эджа нет). То есть ферма форвардит
заведомо забракованные идеи — телега впереди лошади. Это тратит внимание (отладка
пре-маркетных 400, заморозок grid) без отдачи.

## Решение

Ферма переходит в режим бенчмарков: активны только `buyhold`/`random` — живой замер
реального исполнения песочницы, который бэктестер дать не может. Кандидаты отключаются
**мягко** (вариант A): код стратегий и sandbox-счета остаются, перестаёт вызываться тик.

## Изменения

### 1. Точка правды `ACTIVE` отдельно от `REGISTRY` (`lab/strategies.py`)

`REGISTRY` остаётся полным каталогом (нужен отчёту и как код). Вводится явный кортеж:

```python
ACTIVE = ("buyhold", "random")  # режим бенчмарков: кандидаты не прошли study (DSR=0),
                                # код сохранён, форвардить нечего до прохождения бэктеста
```

Вернуть кандидата в работу = дописать его имя в `ACTIVE`.

### 2. Раннер тикает только активных (`lab/runner.py`)

Строка 77: `names = list(REGISTRY)` → `names = list(ACTIVE)`.
Импорт `ACTIVE` рядом с `REGISTRY`. Прочих изменений в цикле нет.

### 3. Финальный снапшот кандидатов (`analysis/farm_candidates_final.py`, новый, вариант B)

Разовый скрипт: читает `lab/lab.db` (журнал фермы), по каждому кандидату выводит
P&L % / maxDD % / число сделок / ошибок и сверяет с предсказанием study (DSR сказал
«эджа нет» — подтвердил ли форвард?). Артефакт в `analysis/`, закрывает их историю.
Запускается с данными VPS (скопировать `lab/lab.db` локально либо прогнать на сервере).

### 4. Отчёт (`lab/report.py`)

- Шапка помечает, кто в `ACTIVE` (живой бенчмарк), а кто архивный кандидат —
  чтобы замёрзшие цифры не читались как баг.
- **Побочный баг-фикс (строка 29):** счёт ошибок собран как
  `strategy=? AND kind LIKE '%error%' OR strategy=? AND kind LIKE '%fail%'` без скобок.
  Из-за приоритета `OR` над `AND` условие разваливается (вторая ветка игнорирует
  strategy-фильтр). Обернуть в скобки: `strategy=? AND (kind LIKE '%error%' OR kind LIKE '%fail%')`.

### 5. Деплой и память

- scp-зеркало `strategies.py` + `runner.py` + `report.py` на VPS → рестарт
  `tinvest-lab.service` → проверка `is-active` + что тикают только 2 контроля.
- Память: записать решение и причину, чтобы будущая сессия не «оживила» кандидатов.

## Вне рамок (YAGNI)

- Счета кандидатов НЕ закрываются (выбор A — мягкое отключение).
- Монитор здоровья фермы НЕ строится: при двух простых контролях следить почти не за чем.
- Код кандидатов НЕ удаляется — рабочая заготовка на случай новой идеи, прошедшей бэктест.
