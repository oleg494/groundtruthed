# T-Invest API — подводные камни и лайфхаки

> Концентрат практического опыта из community-чата T-Invest API (год обсуждений,
> янв 2025 – апр 2026). То, чего нет в официальной документации. Источник —
> NotebookLM по архиву чата. Проверяй актуальность: лимиты и баги меняются.

## 🔢 Цены и Quotation
- Формула строго: `value = units + nano / 1_000_000_000`. Не «склеивать» строками —
  теряются лидирующие нули, `9.066` читается как `9.66`.
- Используй `quotation_to_decimal()` / `money_to_decimal()` из SDK.
- НЕ работать с `float`. Хранить в `DECIMAL(14,6)` или в целых «тиках».
  `13.53 - 0.01` на float = `13.519999...` → ошибки округления к шагу цены.
- Облигации — цена в ПРОЦЕНТАХ от номинала, не в рублях. `GetLastPrices` → `114.28`,
  не `1142.8`. Стоп по облигации тоже в процентах. Пересчёт: `% = рубли / (100 * номинал)`.
- Баг типизации: в `GetPortfolio` поле `quantity` имеет тип `Quotation` (хотя это штуки).

## 📡 Стримы (gRPC/WebSocket)
- Рвётся через 2 минуты без активных подписок. Встроенного реконнекта НЕТ.
- Паттерн: `while True` + `try/except` → при разрыве ПОЛНОСТЬЮ пересоздать объект
  стрима, пауза, переподписаться на всё заново.
- «Тихий» обрыв (соединение живо, данных нет): хранить время последнего сообщения;
  тишина >5–10 сек → дёрнуть `GetMySubscriptions`, нет ответа → форсированный реконнект.
- Пинг: `PingDelaySettings` / `ping_delay_ms` (в одной версии Python SDK баг — поле
  называлось `ping_delay_millis`).
- Свечи в стриме: флаг `waiting_close=false` → свеча в конце каждой минуты
  (за отдельную минуту её может не быть).

## 🚦 Лимиты (актуальные на ~2026)
- 50 rps с одного IP. Превышение = бан на сетевом железе. Авторазбан ДОЛЬШЕ,
  чем сброс `ratelimit_reset`.
- `PostOrder`: 15 заявок/сек (отсчёт от начала секунды). На `PostOrderAsync` НЕ влияет.
- Стримы на клиента (с релиза 1.48): котировки 32, ордера 16, операции 11.
  До 300 подписок в одном стриме.
- Лимиты считаются на токен / все аккаунты разом, не на инструмент.
- Метадата ответа: `x-ratelimit-limit` / `-remaining` / `-reset`.
- Коды: `80001`/`80002` → читать `reset`, делать `sleep`. `80006 ResourceExhausted` =
  circuit breaker (много твоих вызовов за минуту упало с ошибкой).
- Гнать унарные запросы через свой RateLimiter; массовая докачка — оркестратор (Airflow).

## 🆔 Идентификация инструментов
- FIGI — deprecated. У опционов его нет. Может СОВПАДАТЬ у разных фьючерсов
  (`$BRQ6` и `$BRU6` → один FIGI). Не использовать.
- Использовать `instrumentUid` или `ticker_classCode` (можно строкой в `instrument_id`:
  `"SBER_TQBR"`, `"BRZ5_SPBFUT"`).
- UID «меняется» — это фича: корпсобытия (сплиты, ребренд Тинькофф→Т-Банк), переезд
  на другую площадку. НЕ хардкодить UID — на старте `FindInstrument(api_trade_available_flag=true)`
  и обновлять локальный справочник.

## 🛑 Ордера и стоп-заявки
- Стоп-заявки через API ненадёжны — отслеживать цену самому + рыночный ордер.
- Маржиналка (новые счета, требование ЦБ): нужен флаг `confirm_margin_trade: true`
  (`confirmMarginTrade` в REST), иначе отклонение (`30049`, `70002`).
- `GetMaxLots` по инструменту — самый честный способ узнать доступную маржу.
- Баг: `CancelOrder` параллельно исполнившейся заявки → `50005 NOT_FOUND`.
- По опционам — только лимитные заявки, рыночные недоступны.
- `order_id` в `post_order`: либо пусто, либо строго UUID. Timestamp/произвольная строка
  → `INVALID_ARGUMENT`.

## 📦 История и операции
- `GetOperations` — deprecated, возвращает пустые массивы. → `GetOperationsByCursor`.
- НО у курсорного метода баг фильтра `State=Executed` — теряет часть исполненных
  операций (и `margin_fee`). Лайфхак: тянуть БЕЗ фильтра, фильтровать локально.
- **Курсор не листает назад** (проверено 2026-06): `nextCursor` отдаёт ТУ ЖЕ страницу
  (overlap 100/100, new 0), а `limit>100` игнорируется (потолок 100/стр). Полагаться на
  пагинацию нельзя — выгребет только свежие 100. Обход: рекурсивно нарезать окно
  `[from,to]` по времени (окно вернуло ≥100 и `hasNext` → делить пополам). Реализация:
  `analysis/account_reconcile.py::fetch_ops`.
- **Реконсиляция счёта** (`analysis/account_reconcile.py`): две жёсткие тождественности
  против оракула `GetPositions` — (1) Σ payment всех операций == текущий кэш; (2) Σ(BUY−SELL)
  лотов == баланс позиции. uid инструмента МИГРИРУЕТ (один TMON жил под 3 uid одновременно:
  портфель/позиции/операция) — агрегировать по ISIN, не по uid. Основной счёт сходится до
  копейки; на Инвесткопилке остаётся ~1.79 ₽ — операция, не отданная API (см. потерю выше).
- `GetCandles` глубоко в прошлое не отдаёт минутки. Архивы свечей/сделок напрямую:
  `developer.tbank.ru/invest/services/history-md/...` (`.csv.gz`, бесплатно, с 2018).

## ⏰ Сессии / клиринг / выходные
- По выходным (~05:13) в стрим стаканов летят битые `limit_up`/`limit_down`
  → заявка по нормальной цене = `30099` «price outside the limits».
- После вечерней сессии биржа отменяет неисполненные заявки — утром перевыставлять
  (неисполненные могут и «перевыпуститься» — поведение биржи).
- Вармаржу по закрытой внутри дня фьючерсной позиции не узнать до клиринга (~00:00,
  расчётная цена фиксируется в 19:00) — позиция пропадает из `GetPortfolio`.
- Внебиржевые (дилерские) свечи могут попадаться за выходные.

## 📊 Индикаторы
- `GetTechAnalysis` (RSI/MACD/EMA/SMA/BB) НЕ совпадает с TradingView — у Т-Банка свои
  конвенции. НО они вскрыты и воспроизводятся БИТ-В-БИТ локально (2463/2464 значений до
  6-го знака, см. `analysis/techanalysis_validate.py`):
  - **RSI** — сглаживание Уайлдера (`avg=(prev·(N−1)+cur)/N`), не SMA-сглаживание;
  - **BB** — дисперсия ПОПУЛЯЦИОННАЯ (÷N), не выборочная (÷N−1);
  - **EMA/MACD** — α=2/(N+1); seed не важен при достаточном прогреве (затухает как (1−α)ⁿ);
  - индикатор отдаёт значение уже на ПЕРВОМ баре окна — считает по истории ДО него, значит
    для своей реконструкции нужен прогрев (EMA/RSI ≥ сотни баров, SMA/BB ровно N−1).

## 🧮 НКД и доходность облигаций
- **Доходность (`INSTRUMENT_VALUE_YIELD`)** воспроизводится как эффективная годовая, ACT/365,
  расчёты T+1 (сходится с API в пределах ~1.3 б.п., остаток — разный снимок цены; см.
  `analysis/bond_ytm_validate.py`).
- **Дневной НКД из `GetAccruedInterests` НЕ наивно-линейный.** Ряд привязан к дате расчётов
  (T+1) и переносу через выходные/праздники: значения ЗАМОРАЖИВАЮТСЯ на сб/вс (напр. 26238:
  30–31.05 и 01.06 все = 35.21) и дают нерегулярные скачки (04→05.06 на +3 дня). Не считать
  НКД «в лоб» как `C·(d−start)/period` — брать у API на нужную дату расчётов.
- **`GetBondEvents` без `from/to` отдаёт события только за ~ближайший год.** У амортизируемой
  облигации это ОБРЕЗАЕТ график погашений: `RU000A10BNM4` (uid `e551ee93-…`) с дефолтным
  диапазоном отдаёт 2 события `EVENT_TYPE_MTY` (2027-02-12, 2027-05-13) — первое легко принять
  за погашение и вдвое занизить срок. С явным широким `from/to` их 6, финальное — **2028-05-07**
  (сходится с сайтом T-Bank). Для амортов дата погашения = ПОСЛЕДНЕЕ событие MTY, поэтому под
  YTM/дюрацию всегда передавать явный `to` за горизонт бумаги. Проверено live MCP 2026-07-01,
  подтверждено саппортом (community-чат #137370). Воспроизвести: `GetBondEvents(uid, MTY)` без
  дат vs с `to=2028-12-31` → 2 против 6 событий.

## 🕯 Агрегация свечей (проверено, `analysis/candle_aggregate_validate.py`)
- Старший ТФ = точная OHLCV-свёртка младшего: неделя 55/55, месяц 11/11, дневные из часовых
  на регулярных сессиях 12/12 — бит-в-бит. Граница дня/недели/месяца — по MSK (не UTC).
- **Выходные/праздничные ДНЕВНЫЕ свечи MOEX строит отдельным пайплайном** — из часового
  потока не собираются (Δvol до 86k). Для weekend-сессий брать дневную свечу у API напрямую.
- `volumeBuy + volumeSell` НЕ всегда == `volume` у АКЦИЙ: на редких аукционных днях vol чуть
  больше (неклассифицированный аукционный объём, ~0.03–1.6%; общие даты у разных бумаг). На
  ФЬЮЧЕРСАХ сходится всегда. H/L-инвариант (low=min, high=max) и положительность цен — абсолютны.
- Минутная свеча = свёртка обезличенных сделок (`GetLastTrades`): O/H/L/V воспроизводятся
  бит-в-бит, **объём в ШТУКАХ 1:1** (не лотах). НО `close` (24/29) и изредка `open` не
  восстановить однозначно — метки сделок секундные, субсекундный порядок не отдаётся, а в
  последней секунде минуты бывает несколько сделок. Для точного close брать свечу, не свёртку.

## 🎯 Опционы FORTS (проверено, `analysis/options_parity_validate.py`)
- Опционы на спот (напр. Si) — европейские с премией вперёд ⇒ паритет С ДИСКОНТОМ:
  `C−P = D·(F−K)`, поэтому `K+C−P` ЛИНЕЙНА по страйку (не константа). Из наклона достаётся
  дисконт-фактор → рублёвая ставка (≈13.6%, близко к ключевой) и форвард (сходится с фьючерсом).
- `THEOR_PRICE` доступна даже без сделок и безарбитражна (паритет R²=0.999996, бабочка≥0,
  IV колла==IV пута) — удобный источник для проверки котировок и снятия implied-ставки/смайла.

## 💼 Маржа и ГО
- **Риск-ставки `GetRiskRates` для биржевых фондов (TMON@ и др.) ПУСТЫЕ** — начальную маржу
  по ним из публикуемой ставки не пересчитать, брать `GetMarginAttributes` (проверено,
  `analysis/portfolio_margin_validate.py`). Тождества маржи: minimalMargin=½·startingMargin,
  amountOfMissingFunds=startingMargin−liquidPortfolio (до копейки).
- **ГО фьючерса ≠ нотионал × dlongClient.** Проверка по Si: GO/нотионал ≈15.2–15.3% у обоих
  контрактов, а `dlongClient` различается (9.35% у SiU6, 16.65% у SiZ6) — поля
  `dlongClient/dshortClient` из FutureBy НЕ являются ставкой ГО. Брать ГО прямо из
  `GetFuturesMargin` (`initialMarginOnBuy/Sell`). Нотионал Si = price/minPriceIncrement ×
  minPriceIncrementAmount (= цена×1000_USD в рублях).

## 🐷 Инвесткопилка
- Сделки Инвесткопилки идут через ВНЕБИРЖЕВОЙ инструмент `TMON OTC` «Денежный рынок»
  (kind=ETF, uid `cd03cec6-…`), у которого НЕТ биржевых свечей (`GetCandles`=0). Это НЕ тот же
  uid, что биржевой TMON@ (`498ec3ff-…`). Поэтому: цену сделок копилки нельзя сверять с биржевыми
  свечами (это OTC-котировка), GetMaxLots по нему падает с INTERNAL, GetRiskRates пуст.
  Сверка «цена сделки ∈ [low,high] свечи» на биржевом счёте — 67/67, на копилке неприменима.
- `FindInstrument("TMON")` отдаёт **6 листингов на один ISIN** (RU000A106DL2): классы
  TQBR/TQTF/PTEQ/PTTF/OTC + `TMON@`/SPBRU. Через API торгуется ТОЛЬКО `TMON@` (SPBRU),
  `apiTradeAvailableFlag=true`; у остальных пяти — `false` (проверено live 2026-07-01). Для
  market-data брать `498ec3ff-…` (SPBRU) — только у него биржевые свечи; агрегировать по ISIN.

## 📦 Архивы History MD (history-data) — объём в лотах ИЛИ штуках (ненадёжно)
- Годовой zip минутных свечей: `GET https://invest-public-api.tinkoff.ru/history-data?instrumentId={uid}&year={YYYY}`
  с Bearer-токеном. Внутри — по CSV на торговый день, `;`-разделитель, без заголовка:
  `UID;ts(UTC);open;close;high;low;volume;`. Лимит — 30 файлов/мин (отдельный).
- **OHLC бит-в-бит совпадают с `GetCandles`** (проверено оракулом, `analysis/history_archive_validate.py`),
  набор минут один-в-один — архив пригоден как офлайн-источник котировок.
- **ГРАБЛЯ: колонка `volume` масштабирована непоследовательно.** У SBER до **2025-07-31**
  включительно объём в ЛОТАХ (api/archive ×10), с 2025-07-31 — в штуках (×1); у GAZP/LKOH ×1
  уже раньше. Это НЕ универсальное «архив в лотах» — масштаб зависит от инструмента/периода
  (вероятная причина у SBER — смена размера лота 10→1 в истории). Объём из архива как абсолют
  не использовать: сверять с `GetCandles` или брать только относительные профили внутри файла.
- Host `*.tbank.ru` для history-data отдаёт self-signed Минцифры-серт (нужен `SSL_TBANK_VERIFY`);
  старый `*.tinkoff.ru` работает с системными CA — REST-слой проекта на нём.

## 🔒 SSL / сертификаты
- Сервер использует цепочку Russian Trusted Root CA (Минцифры): `*.tbank.ru` ←
  `Russian Trusted Sub CA` ← `Russian Trusted Root CA`.
- Node/curl/Python/JDK по умолчанию не доверяют → `SELF_SIGNED_CERT_IN_CHAIN` / cURL (60).
- Решение: установить корневой CA Минцифры (для Node — `NODE_EXTRA_CA_CERTS`).
- Бандл: https://gu-st.ru/content/Other/doc/russiantrustedca.pem
